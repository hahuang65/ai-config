API Rate Limiting

The Acme API had zero protection against abusive traffic. Any client — authenticated or not — could make unlimited requests, risking resource exhaustion and degraded service for all users. This change adds a Redis-backed sliding window rate limiter that runs after authentication, applies plan-based limits (100/1k/10k per minute), and fails open on Redis outages to avoid cascading failures.

Code Review

Good

  • Atomic Redis pipeline prevents race conditions under concurrent load
  • Fail-open design — Redis outage doesn't cascade into API outage
  • Plan-based limits are configurable via environment variables
  • Per-route override factory integrates cleanly with FastAPI's dependency injection
  • Response headers (X-RateLimit-*) set on every request, not just rate-limited ones

Bad

  • None found

Ugly

  • str(now) as sorted set member creates unbounded key growth if requests arrive at the exact same timestamp (microsecond collision). Consider using f"{now}:{uuid4().hex[:8]}" for uniqueness.
  • Sliding window log uses O(n) memory per client per window. At enterprise tier (10k/min), this is ~500KB per active client. Acceptable now but worth monitoring.

Questions

  • Should rate limit headers be set on OPTIONS preflight requests? Currently they are, which adds unnecessary Redis calls for CORS preflight.
  • Should there be an admin API for querying/resetting a client's rate limit window?
Decision Log
High Confidence

Sliding window log over fixed window

Rationale: Fixed windows allow burst traffic at boundaries (up to 2x the limit). Sliding window log provides smooth, accurate limiting with no boundary spikes. The O(n) storage cost is acceptable for our traffic levels.

Alternatives rejected: Fixed window (boundary burst issue), token bucket (harder to implement correctly with distributed state), leaky bucket (doesn't support burst capacity at all).

High Confidence

Fail-open on Redis outage

Rationale: A Redis outage should not cascade into a full API outage. The risk of excess traffic during Redis downtime (~minutes) is acceptable and can be mitigated at the ALB level. Fail-closed would have made Redis a hard dependency for all API traffic.

Medium — Inferred

Custom implementation over slowapi

Rationale: The existing codebase uses async redis-py directly. A custom ~60-line implementation avoids a library dependency, gives full control over key format and plan-based logic, and integrates cleanly with the existing dependency injection pattern.

Trade-off: More code to maintain, but the algorithm is simple and well-tested. Revisit if requirements grow significantly (e.g., multi-tier rate limiting, dynamic rules).

Re-entry Context (for future you)

Key invariants

  • The Redis pipeline MUST use transaction=True. Without it, concurrent requests can read stale counts.
  • Rate limit dependency MUST run after auth dependency (it reads user.plan).

Non-obvious coupling

  • The redis_key_prefix in config.py is shared between session storage, caching, and rate limiting. Changing it affects all three.

Gotchas

  • fakeredis doesn't perfectly emulate PEXPIRE timing — integration tests with real Redis may behave slightly differently for edge cases around window expiration.
