Key Implementation — Sliding Window
src/dependencies/rate_limit.py — core algorithm
async def check_rate_limit(
    request: Request, response: Response,
    user: User = Depends(get_current_user),
    redis: Redis = Depends(get_redis),
) -> None:
    key_id = user.api_key_id or user.id or request.client.host
    window = settings.rate_limit_window_seconds
    limit = RATE_LIMITS.get(user.plan, settings.rate_limit_default)
    redis_key = f"{settings.redis_key_prefix}ratelimit:{key_id}:{window}"
    now = time.time()

    try:
        async with redis.pipeline(transaction=True) as pipe:
            pipe.zremrangebyscore(redis_key, 0, now - window)
            pipe.zadd(redis_key, {str(now): now})
            pipe.zcard(redis_key)
            pipe.pexpire(redis_key, window * 1000)
            results = await pipe.execute()

        count = results[2]
        remaining = max(0, limit - count)
        response.headers["X-RateLimit-Limit"] = str(limit)
        response.headers["X-RateLimit-Remaining"] = str(remaining)
        response.headers["X-RateLimit-Reset"] = str(int(now + window))

        if count > limit:
            raise RateLimitExceeded(retry_after=window, limit=limit)
    except (ConnectionError, TimeoutError):
        pass  # fail open

Implementation Notes
Backward compatibility. Adding the rate limit dependency globally via app = FastAPI(dependencies=[Depends(check_rate_limit)]) applies to all routes. Existing clients with normal usage will not be affected — free plan allows 100 req/min, which is above typical usage patterns.
Critical: atomic operations. The Redis pipeline MUST use transaction=True to ensure ZREMRANGEBYSCORE + ZADD + ZCARD execute atomically. Without this, concurrent requests can read stale counts and exceed the limit.
Performance consideration. Each rate-limited request adds one sorted set member (~50 bytes) to Redis. At peak traffic (10k enterprise requests/min/client), the worst case is ~500KB per active client per window. Keys auto-expire via PEXPIRE, so storage is bounded.
