Process Steps
1 Overview

The implement skill consumes an approved plan.md from docs/claude/ and executes every task in its todo list, in order. It marks tasks as completed, runs type checks and linters continuously, and never stops to ask for permission between tasks. The plan is the spec — follow it faithfully.

Implementation should be "boring." If implementation surprises you, the plan wasn't detailed enough. All creative decisions were made during planning. This phase is pure execution.
2 Prerequisites
Required Before Starting

  • The user has explicitly approved the plan — never assume approval
  • User provides the directory path, slug/partial path, or no argument (resolves most recent)

3 The 8-Step Process

Read the Plan

Read the plan document thoroughly. Understand every task, every code snippet, every constraint before writing a single line.

Implement Everything

Execute all tasks in the todo list, in order. Use test-driven development. Mark each task - [x] as completed. Run type checks and linters continuously. Do not stop to ask for confirmation between tasks.

Track Progress

Update the plan document after completing each task or phase. The plan document is the source of truth for what is done and what remains.

Maintain Code Quality

Follow existing code patterns and conventions. Every change must have test coverage. Maintain strict typing — no any or unknown. No unnecessary comments or documentation unless the plan says to.

Verify Tests

After all tasks are done, run the full test suite (or the relevant subset) and confirm all tests pass. If any tests fail, fix them before declaring completion.

Invoke /simplify

Review the changed code for reuse opportunities, quality issues, and efficiency improvements. Fix any issues found. Then re-run the test suite to confirm nothing broke.

Invoke /fact-check

Fact-check the plan document. Verify that all claims — file paths, line numbers, function names, behavior descriptions — match what was actually implemented. This is not optional.

Report Completion

Tell the user implementation is complete and summarize what was done, including test coverage added. Do NOT commit to version control — leave that to the user.

4 TDD: Non-Negotiable

Test-Driven Development Is Required

Write tests before or alongside implementation code. If the plan's todo list is missing test tasks, write tests anyway. Every behavioral change must have test coverage. The absence of test tasks in the plan does not excuse the absence of tests in the implementation.

Hard requirement, not a suggestion. Each change MUST have test coverage. If the plan says to do something, do it. If the plan doesn't mention something, don't do it — with one exception: tests are always required, even if the plan omits them.
5 Test Structure
Shared Setup
Maximize shared setup. Use before, let, subject, factory blocks (or the codebase's equivalent) so common state is defined once. No duplicated setup across tests.
Validate Setup
Write a test that validates the shared setup is correct. This baseline test ensures the foundation is solid before testing mutations on top of it.
Minimal Mutations
Each individual test should be minimal: load the shared setup, apply the bare minimum mutation for the scenario, and assert. Keep tests focused and small.
6 Code Quality
Standards
  • Follow existing code patterns and conventions in the codebase
  • Maintain strict typing — avoid any or unknown types
  • Do not add unnecessary comments or documentation unless the plan says to
  • Keep code clean and consistent with surrounding code
  • When the user references existing code, read that reference code first and match it precisely
  • Most features in a mature codebase are variations on existing patterns
✘
NEVER commit to version control. No git add, no git commit, no git push. The user will commit when they are ready. This is a hard rule with no exceptions.
7 Issue Handling
Minor
Fix and continue
Note the deviation in the plan document. Do not stop for trivial issues.
Significant
Stop and report
If something in the plan cannot be implemented as written, stop and tell the user what the issue is. Wait for guidance before continuing.
Test Failure
Fix if clear, otherwise stop
If the cause of the test failure is obvious, fix it and move on. If not, stop and report the failure to the user.
8 Feedback Loop
Post-Implementation Corrections

After implementation, the user may test and provide terse corrections. These will be short and direct. Act on them immediately without asking for clarification — you have full context from the plan and session.

  • "wider" — act immediately, you have the context
  • "still cropped" — you saw the prior attempt, fix it
  • "move this to the admin app" — relocate without asking why
When the user reverts your changes: Do not try to patch a bad approach. Start fresh with the narrowed scope the user provides. A clean restart almost always produces better results than incremental fixes.
Referencing Existing Code

When the user references existing code — "make it look like the users table", "same pattern as the auth middleware" — read that reference code first and match it precisely. Most features in a mature codebase are variations on existing patterns.

9 Important Guidelines
Do
  • Follow the plan faithfully — it is the spec
  • Use test-driven development for every change
  • Run type checks and linters continuously
  • Mark tasks - [x] as you complete them
  • Match existing code patterns precisely
  • Invoke /simplify after all tasks are done
  • Invoke /fact-check on the plan document
  • Summarize what was done including test coverage
Do Not
  • Stop between tasks to ask for permission
  • Add features or improvements not in the plan
  • Refactor code that is not part of the plan
  • Add unnecessary comments or documentation
  • Use any or unknown types
  • Commit to version control — ever
  • Patch a bad approach — start fresh instead
  • Skip tests, even if the plan omits them
The plan is the spec. If the plan says to do something, do it. If it doesn't mention something, don't do it. Implementation is execution, not design. The one exception: tests are always required.
Visual Companion

When invoked as part of the build-feature workflow, the orchestrator may generate a visual diff review after all implementation tasks are complete — but only if the visual-explainer skill is available. The implement skill itself does not generate the review; it focuses purely on executing the plan.
