02 The Big Checklist — 14 Steps

Step Task Agent / Command Required? Fix-if-fail? Observable Output
1 Read the plan thoroughly — understand every task, snippet, constraint — MUST — Plan document loaded
2 Implement everything via TDD cycle. Mark tasks [x]. Run checks continuously. Write tests even if plan omits them. tdd-guide MUST Yes Source files modified, tests passing, plan tasks checked
3 Track progress — update plan.md after each task/phase — MUST — plan.md reflects current state
4 Maintain code quality — follow existing patterns, no unnecessary comments — MUST — Code matches codebase conventions
5 Verify (comprehensive) — Type check + Lint + Test + Build. Repeat until all pass. — MUST Yes All 4 checks pass cleanly
6 Security review — fix CRITICAL immediately. Reports CRITICAL / HIGH / MEDIUM. security-reviewer MUST Yes (CRITICAL) Security findings report
7 Database review — only if DB code touched. Fix CRITICAL / HIGH. database-reviewer COND Yes (CRIT+HIGH) DB review findings
8 Simplify — review for reuse, quality, efficiency. Re-run tests after. /simplify MUST Yes Simplified code, tests still pass
9 Refactor cleanup — remove SAFE items, verify CAREFUL items. Re-run tests. refactor-cleaner MUST Yes Dead code removed, tests pass
10 Code review — >80% confidence threshold. Fix CRITICAL / HIGH. Re-run tests. code-reviewer MUST Yes (CRIT+HIGH) Code review report, issues resolved
11 Documentation update — if new features, API changes, or architecture mods doc-updater COND — Updated docs
12 Fact-check plan.md — verify all claims match actual implementation /fact-check MUST Yes Corrections applied to plan.md
13 Refresh visual-plan.html — if it exists AND fact-check corrected plan, deviations noted, or tasks added/removed. Last step before reporting. /generate-visual-plan COND — visual-plan.html regenerated
14 Complete — summarize what was done including test coverage. NEVER commit. — MUST — Summary to user, no git operations

03 Agent Registry

tdd-guide
Step 2 Sonnet MUST

Guides red-green-refactor TDD cycle for each task. Ensures tests are written for every behavioral change, even when plan omits test tasks.

security-reviewer
Step 6 Sonnet MUST

Reviews all changed files for security issues. Reports CRITICAL / HIGH / MEDIUM severity. CRITICAL must be fixed immediately.

database-reviewer
Step 7 Sonnet COND

Reviews SQL queries, migrations, schema changes, ORM operations. Only invoked if feature touches database code. Fix CRITICAL and HIGH.

refactor-cleaner
Step 9 Sonnet MUST

Identifies dead code, unused imports, unnecessary abstractions. Items marked SAFE are removed; CAREFUL items are verified before removal. Re-run tests after.

code-reviewer
Step 10 Sonnet MUST

Reads and enforces project rules/ files. Applies confidence-based filtering (>80% threshold). Reports findings by severity. Fix CRITICAL and HIGH.

doc-updater
Step 11 Sonnet COND

Updates documentation when implementation adds new features, changes APIs, or modifies architecture. Skipped for trivial changes.

05 Verification Loop Flow

Step 5 runs this loop until all four checks pass cleanly. Failures at any stage loop back for fixes.

    TEST -->|Pass| BUILD{"Build"}
    TEST -->|Fail| FIX3["Fix test failures"] --> TC

06 Error Handling Rules

  • Minor issues — Fix and continue. Note the deviation in plan.md.
  • Significant deviations — If something in the plan cannot be implemented as written, STOP. Tell the user. Wait for guidance before continuing.
  • Test failures — Fix if the cause is clear. If not, stop and report.
  • NEVER commit — No git add, git commit, or git push. The user commits when ready.

07 Post-Implementation (from build-feature)

When /implement is invoked as Phase 3 of /build-feature, the orchestrator adds these steps after completion:

  • /diff-review — Visual diff review generating an HTML page with executive summary, KPI dashboard, architecture comparison, code review analysis, and decision log (requires visual-explainer skill)
  • /fact-check — Additional fact-check pass on the plan document from the build-feature orchestrator level

The implement skill itself does NOT generate the visual diff review — it focuses purely on executing the plan. The orchestrator handles visual artifacts when the visual-explainer skill is available.
