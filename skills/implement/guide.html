1 overview

The implement skill consumes an approved plan.md and executes every task in its todo list, in order. It marks tasks as completed in the plan document as it goes, runs type checks and linters continuously, and does not stop to ask for permission between tasks. The plan is the spec — follow it faithfully.

Implementation should be "boring". All creative decisions were made during planning. If implementation surprises you, the plan wasn't detailed enough. Go back to planning.

3 process

Read the plan

Read the plan document thoroughly. Understand every task, every code snippet, every constraint.

Implement everything

Execute all tasks in the todo list, in order. Use test-driven development. Mark each task - [x] as completed. Run type checks and linters continuously.

Track progress

Update the plan document after each task. The plan is the source of truth for what's done and what remains.

Maintain quality

Follow existing code patterns. Significant test coverage. Strict typing — no any or unknown. No unnecessary comments. Clean and consistent.

Complete

After all tasks are done, summarize what was implemented. When invoked from build-feature, a diff review HTML is generated automatically.

4 issue handling
MINOR
Fix and continue
Note the deviation in the plan. Don't stop for trivial issues.
SIGNIFICANT
Stop and report
If something can't be implemented as written, stop and tell the user. Wait for guidance.
TEST FAIL
Fix if clear, otherwise stop
If the cause is obvious, fix it. If not, stop and report the failure.
5 feedback loop
Post-implementation corrections

After implementation, the user may test and provide terse corrections. These will be short and direct. Act on them immediately without asking for clarification.

  • "wider" — act immediately, you have full context
  • "still cropped" — the agent saw the prior attempt, fix it
  • "move this to the admin app" — relocate without asking why
When the user reverts your changes: Do not try to patch a bad approach. Start fresh with the narrowed scope the user provides. A clean restart almost always produces better results than incremental fixes.
6 rules
Do
  • Follow the plan faithfully
  • Use test-driven development
  • Run type checks continuously
  • Mark tasks complete as you go
  • Match existing code patterns exactly
  • Reference existing code when told to
Don't
  • Stop between tasks to ask permission
  • Add features not in the plan
  • Refactor code outside the plan scope
  • Add unnecessary comments or docs
  • Use any or unknown types
  • Patch a bad approach — restart fresh
The plan is the spec. If the plan says to do something, do it. If it doesn't mention something, don't do it. Implementation is execution, not design.
