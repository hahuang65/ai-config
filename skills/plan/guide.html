1 Overview

The plan skill creates a detailed plan.md — a real markdown file, not a chat summary. It includes approach, file-by-file changes with real code snippets, testing strategy, and trade-offs. You then annotate it with corrections, and the agent addresses every note. This cycle repeats until you explicitly approve. Only then does implementation begin.

Never write code until the plan is reviewed and approved. This is the most important guard rail. All creative decisions happen here, not during implementation.
2 Core Principle
The plan is a shared mutable document.

You inject judgment through annotations: domain knowledge, business constraints, engineering trade-offs. The agent does the research and writing, you steer. Each cycle makes the plan more precise. By the time you approve, implementation should be boring — just following the spec.

3 Plan Contents

Goal

What we're building and why. 1-2 sentences establishing the target.

Research Reference

Path to the research document this plan is based on. Grounds the plan in verified understanding.

Approach

High-level strategy and architectural decisions. The "why" behind the design choices.

Detailed Changes

For each file: path, what changes are needed, why, and actual code snippets (not pseudocode).

New Files

Any new files to create, with their purpose and structure.

Dependencies

New libraries or services needed.

Considerations & Trade-offs

Alternative approaches considered and why this one was chosen.

Migration / Data Changes

Database migrations, data backfills, config changes.

Testing Strategy

What tests to write or update. Coverage targets and edge cases to verify.

4 Annotation Cycle
How annotations work

Open plan.md in your editor. Add inline notes anywhere — correct assumptions, reject approaches, add constraints, provide domain knowledge. Then tell the agent to address your notes. It reads the file, finds every annotation, addresses them, updates the plan, and removes the notes. Repeat until satisfied.

  • Correct "this should be a PATCH, not a PUT"
  • Reject "remove this section entirely, we don't need caching here"
  • Constrain "the signatures of these functions should not change"
  • Domain "use drizzle:generate for migrations, not raw SQL"
Every annotation must be addressed. The agent never ignores feedback. If it disagrees with a note, it should explain why and propose an alternative — not silently skip it.
5 Todo List
After approval

When you approve the plan, the agent appends a detailed task checklist. Each task is a single, clear unit of work, grouped into logical phases. This checklist becomes the implementation tracker — tasks are checked off as they're completed.

## Todo List
### Phase 1: Database
- [ ] Create migration for tokens table
- [ ] Add index on user_id column
### Phase 2: API Layer
- [ ] Add POST /tokens endpoint
- [ ] Add GET /tokens/:id endpoint
- [ ] Add DELETE /tokens/:id endpoint
### Phase 3: Tests
- [ ] Unit tests for token service
- [ ] Integration tests for API endpoints
Still no code. Even after the todo list is generated, wait for the user to explicitly trigger implementation. The plan phase produces the spec; implementation consumes it.

7 Quick Start

Invoke planning

Type /plan add cursor-based pagination to users API

Agent writes the plan

It reads any existing research, then produces plan.md with real code snippets and file-by-file changes.

Annotate in your editor

Open plan.md, add inline notes. Tell the agent to address them. Repeat 1-6 times.

Approve

Say "approved" or "looks good". The agent generates a todo list. Now you're ready for /implement.
