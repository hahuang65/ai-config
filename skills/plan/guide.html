Plan

1 Overview

The plan skill produces a real plan.md file -- not a chat summary. It contains the approach, file-by-file changes with actual code snippets, testing strategy, and trade-offs. You then annotate it with // comments to correct assumptions, reject approaches, add constraints, or provide domain knowledge. The agent addresses every note and updates the document. This cycle repeats 1-6 times until you explicitly approve. Only then does a todo list get generated and implementation may begin.

The file IS the specification. Use a real markdown file, not chat summaries. Include actual code snippets, not vague descriptions. Reference existing code patterns from the codebase when proposing changes.
2 Core Principle
Never write code until the plan is reviewed and approved.

The plan is a shared mutable document between you and the agent. You inject judgment through annotations: domain knowledge, business constraints, engineering trade-offs. The agent does the research and writing, you steer. Each cycle makes the plan more precise. By the time you approve, implementation should be boring -- just following the spec.

Always prefer the cleaner, more maintainable approach. When choosing between approaches, favor the one that is simpler to understand, easier to maintain long-term, and produces less technical debt -- even if it requires slightly more upfront effort. Avoid clever shortcuts that trade long-term clarity for short-term convenience.
3 The 6-Step Process

Read Context

Check docs/claude/ for recent feature directories containing a research.md related to the topic. Read relevant source files. Base the plan on the actual codebase, not assumptions.

Input gathering

Write the Plan Document

Create plan.md with all 9 sections: Goal, Research Reference, Approach, Detailed Changes (with real code snippets per file), New Files, Dependencies, Considerations & Trade-offs, Migration/Data Changes, and Testing Strategy.

Output: plan.md

Wait for Annotation

STOP and tell the user the exact file path. Explain the // annotation convention. The user opens the file, adds inline comments anywhere, then tells the agent to address the notes.

Blocked -- waiting for user

Address Annotations

Read the updated plan. Find ALL // annotations. Address every single note -- never skip any. Update the plan accordingly. Remove processed annotations so they don't accumulate. STOP and tell the user the plan is updated. Repeat this cycle until the user approves.

Repeats 1-6 times

Generate Todo List

When the user approves, append a detailed task checklist grouped into logical phases. Each task is a single, clear unit of work. Test tasks are mandatory -- every test case from the Testing Strategy must appear as a - [ ] item.

Output: appended to plan.md

Generate Visual Plan

Invoke /generate-visual-plan to produce an HTML page with state machines, before/after comparisons, file maps, edge cases, and code snippets. Output to visual-plan.html in the same feature directory as plan.md. Open it in the browser.

Output: visual-plan.html
Do NOT implement after Step 6. Even after the todo list and visual plan are generated, wait for the user to explicitly say "implement." The plan phase produces the spec; implementation consumes it.
4 Plan Contents

Goal

What we're building and why. 1-2 sentences establishing the target.

Research Reference

Relative path or name of the research file used to generate the plan. Grounds the plan in verified understanding.

Approach

High-level strategy and architectural decisions. The "why" behind the design choices.

Detailed Changes

For each file: path, what changes are needed, why, and actual code snippets (not pseudocode).

New Files

Any new files to create, with their purpose and structure.

Dependencies

New libraries or services needed.

Considerations & Trade-offs

Alternative approaches considered and why this one was chosen.

Migration / Data Changes

Database migrations, data backfills, or config changes.

Testing Strategy Concrete tests required

Specific test cases -- not vague descriptions. Each must name the test file, describe the scenario, and state the expected outcome. Every test case listed here MUST appear as a task in the Todo List.

5 Annotation Convention
The // inline comment system

Open plan.md in your editor. Add // comments anywhere -- corrections, rejections, constraints, or domain knowledge. Then tell the agent to address your notes. It reads the file, finds every annotation, addresses them, updates the plan, and removes the processed notes so they don't accumulate.

  • Correct // this should be a PATCH, not a PUT
  • Reject // remove this section entirely, we don't need caching here
  • Constrain // the signatures of these functions should not change
  • Domain // use drizzle:generate for migrations, not raw SQL
### `src/api/users.ts` // this should be a PATCH, not a PUT - Update the `updateUser` handler to accept partial updates via PUT // remove this section entirely, we don't need caching here - Add Redis caching layer for user lookups // use drizzle:generate for migrations, not raw SQL
Every annotation must be addressed. The agent never ignores feedback. If it disagrees with a note, it should explain why and propose an alternative -- not silently skip it.
6 Todo List & Tests
After user approves the plan

The agent appends a detailed task checklist to plan.md. Each task is a single, clear unit of work, grouped into logical phases. This checklist becomes the implementation tracker.

## Todo List
### Phase 1: Database
- [ ] Create migration for tokens table
- [ ] Add index on user_id column
### Phase 2: API Layer
- [ ] Add POST /tokens endpoint
- [ ] Add GET /tokens/:id endpoint
- [ ] Add DELETE /tokens/:id endpoint
### Phase 3: Tests
- [ ] Test token creation with valid user
- [ ] Test token lookup returns 404 for missing ID
- [ ] Test deletion is idempotent
Test tasks are mandatory. The Todo List MUST include tasks for every test case from the Testing Strategy. Tests are not optional. Place test tasks in a dedicated phase or interleave them with implementation tasks for TDD ordering (write test before the code it validates).
7 Visual Plan

8 File Naming
Standalone invocation
docs/claude/<YYYYMMDD-HHMM>-<slug>/plan.md

Derive a short slug from arguments (lowercase, hyphens, max ~5 words). Get timestamp via date +%Y%m%d-%H%M. Reuse research directory if one already exists for the same topic.

From build-feature
Orchestrator provides directory path.

Write plan.md into the provided directory. No slug derivation or timestamp needed.

Examples:
docs/claude/20260227-1430-cursor-pagination/plan.md
docs/claude/20260227-1545-sortable-ids/plan.md
9 Important Guidelines
Cleaner approach
Prefer maintainability over cleverness
Always favor the approach that is simpler to understand, easier to maintain long-term, and produces less technical debt -- even if it requires more upfront effort.
Real code
Actual snippets, not pseudocode
Include real code snippets in the plan. Reference existing code patterns in the codebase. If you've seen a good implementation in open source, use it as a model.
Never skip feedback
Every annotation must be addressed
The agent never ignores feedback. If it disagrees with a note, it explains why and proposes an alternative. Silent skipping is not allowed.
Scope control
Actively suggest cutting scope
Keep the plan focused. If scope grows too large, the agent should proactively suggest trimming it down to maintain achievability.
File is the spec
The markdown file IS the specification
Use a real markdown file, not chat summaries. This persistent artifact is the shared source of truth between user and agent.
Tests are mandatory
Testing strategy with concrete cases
The Testing Strategy must include specific test cases -- each naming the test file, scenario, and expected outcome. Every test case must appear in the Todo List.
Quick start: Type /plan add cursor-based pagination to users API. The agent reads existing research, writes plan.md with real code snippets, then waits for your annotations. Add // notes in the file, tell the agent to address them, repeat until satisfied, then say "approved" to get the todo list and visual plan.
