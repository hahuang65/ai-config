Workflow Guide

Build Feature

A disciplined 3-phase workflow for building features with AI assistance. Research deeply, plan meticulously with annotation cycles, then execute the whole thing. Visual HTML companions are generated at each phase transition when the visual-explainer skill is available.

1 Overview

/build-feature orchestrates three sub-skills — research, plan, and implement — into a single disciplined workflow based on Boris Tane's AI coding methodology. The visual-explainer skill is optionally invoked at each phase boundary to produce rich HTML companions (architecture diagrams, visual plans, diff reviews). If visual-explainer is not installed, all visual steps are silently skipped — the workflow proceeds normally with just the markdown artifacts.

Phase 1

Research

Deep-read the codebase. Produce a markdown research doc and an HTML architecture diagram. You review both before moving on.

Phase 2

Plan + Annotate

Write a detailed implementation plan with real code snippets. You annotate, the agent addresses notes. Repeat 1–6 times until you approve.

Phase 3

Implement

Execute every task from the plan. Track progress by checking off items. After completion, a fact-checked diff review HTML is generated.

Optional

Visual Companions

Generated by sub-skills directly via visual-explainer. Self-contained HTML with Mermaid diagrams, CSS Grid layouts, styled tables, and dark/light theme support.

3 Phase 1 — Research
What happens

The research sub-skill reads every relevant file, traces data flows end-to-end, identifies patterns and edge cases, and writes a comprehensive research.md. If visual-explainer is available, it also generates an architecture.html with module dependency diagrams and opens it in your browser.

Your job: Review the research document. Does the agent understand the system correctly? If not, say so and the research will be revised. When satisfied, say "move to planning".
research.md includes
  • › System overview and scope
  • › Architecture and component relationships
  • › Key files with their roles
  • › Data flow traces
  • › Patterns and conventions
  • › Edge cases and gotchas
architecture.html (optional)
  • › Module dependency diagram (Mermaid)
  • › Data flow visualization
  • › Dark/light theme support
  • › Zoom controls on all diagrams
  • › Staggered entrance animations
  • › Opens automatically in browser
4 Phase 2 — Plan + Annotate
What happens

The plan sub-skill writes a detailed plan.md with real code snippets, file-by-file changes, testing strategy, and trade-offs. You open it in your editor, add inline annotations, and the agent addresses every note. This annotation cycle repeats (typically 1–6 times) until you explicitly approve. Then a granular todo list is generated and, if visual-explainer is available, a visual-plan.html with state machines, before/after panels, and edge case tables.

Annotation patterns you can use:
"this should be a PATCH, not a PUT" — correct assumptions
"remove this section entirely" — reject approaches
"the signatures of these functions should not change" — add constraints
"use drizzle:generate for migrations" — provide domain knowledge
No code is written until you approve the plan. This is the most important guard rail in the entire workflow. All creative decisions happen here — not during implementation.
5 Phase 3 — Implement
What happens

The implement sub-skill executes every task from the plan's todo list, checking off items as it goes. It runs type checks and linters continuously. After all tasks are complete, if visual-explainer is available, it generates a diff-review.html comparing the working tree against the branch point, then automatically fact-checks the review (correcting any fabricated numbers or function names) before opening it in your browser.

diff-review.html includes
  • › Executive summary with KPI dashboard
  • › Before/after architecture diagrams
  • › Side-by-side code comparisons
  • › Good / Bad / Ugly code review
  • › Decision log with rationale
  • › Verification summary (fact-checked)
During implementation
  • › Terse corrections are fine: "wider", "still cropped"
  • › Agent acts immediately — context is established
  • › Type checks and linters run continuously
  • › Run /fact-check on any HTML page anytime
6 File Structure
docs/claude/ — one timestamped directory per feature
docs/claude/20260304-1430-cursor-pagination/
  research.md ← Phase 1 deep-read
  architecture.html ← Phase 1 visual (optional)
  plan.md ← Phase 2 annotated plan
  visual-plan.html ← Phase 2 visual (optional)
  diff-review.html ← Phase 3 fact-checked review (optional)
Naming convention: <YYYYMMDD-HHMM>-<slug> — the slug is derived from the feature description (lowercase, hyphens, no special characters, max ~5 words). The directory is created once at the start of the workflow and reused across all three phases.
Cleanup options: After the feature is complete, you can keep the directory for reference, delete it, add docs/claude/ to .gitignore, or commit the directory alongside the feature for posterity.

8 Quick Start

Invoke the workflow

Type /build-feature add cursor-based pagination to the users API

Review research

Read research.md and browse the architecture diagram in your browser (if generated). Confirm it's correct or point out what's wrong.

Say "move to planning"

The agent writes plan.md. Open it in your editor, add inline notes, and tell the agent to address them. Repeat the annotation cycle until satisfied.

Approve the plan

Say "approved" or "looks good". The agent generates a granular todo list and optionally a visual plan HTML.

Say "implement"

The agent executes every task, checks them off, and produces a fact-checked diff review when done. That's it.

9 Key Principles
Never code before approval

The plan must be explicitly approved. This guard rail prevents wasted work on wrong designs.

Markdown files are the deliverables

Not chat summaries. The files in docs/claude/ are the source of truth, readable by anyone.

Visual HTML pages are companions

They provide spatial understanding that markdown cannot. Architecture diagrams, state machines, before/after panels — all generated by sub-skills via visual-explainer.

You inject the judgment

Through annotations: domain knowledge, business constraints, engineering trade-offs. The agent does the work, you steer.

Implementation should be boring

All creative decisions happen during planning. If implementation surprises you, the plan wasn't detailed enough.

Terse corrections are fine

During implementation, the context is already established. One-word feedback like "wider" or "still cropped" is perfectly clear.

10 Session Management

Single Long Session

The entire workflow runs in one session. By the time implementation starts, deep understanding has been built through research and planning. All artifacts — markdown and visual HTML — live together in the feature directory, survive context compaction, and can be re-read at any point.

Context compaction: Because all state is persisted to files in docs/claude/, the workflow is resilient to long conversations. The agent can always re-read the research document, the plan, and the todo list to recover context.
11 Visual Integration
How it works

The visual-explainer skill is optional. All visual steps are skipped gracefully if it is not installed — no errors, no warnings. When available, visual companions are generated by the sub-skills directly (research generates architecture.html, plan generates visual-plan.html, implement generates diff-review.html). Visual-explainer also activates proactively during any phase: when about to render a complex table (4+ rows or 3+ columns) in the terminal, it generates an HTML table instead and opens it in the browser.

Mermaid Diagrams Flowcharts, sequence diagrams, state machines, ER diagrams. Automatic node positioning with deep theme integration.
CSS Grid Layouts Architecture overviews, component maps, system-level diagrams where spatial arrangement matters.
Styled Tables Data comparisons, risk matrices, configuration diffs. Sortable columns, striped rows, responsive scrolling.
Dark / Light Themes Every generated page respects prefers-color-scheme. Both modes are fully designed, not afterthoughts.
Zoom Controls All diagrams have +/- zoom buttons, scroll-to-zoom (Ctrl+wheel), and click-drag panning for large diagrams.
Proactive Rendering Complex tables are intercepted before terminal output and rendered as rich HTML pages instead, opened in the browser automatically.
Fact-checking: After generating any visual output, /fact-check can be run to verify all claims, numbers, function names, and file paths against the actual codebase and git history. This happens automatically for diff reviews during the build-feature workflow.
