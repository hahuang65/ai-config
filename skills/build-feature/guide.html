Workflow Guide

Build Feature

A disciplined 3-phase workflow for building features with AI assistance. Research deeply, plan meticulously, then execute. Visual HTML diagrams are generated at each phase transition.

1 Overview

/build-feature orchestrates three sub-skills — research, plan, and implement — plus the visual-explainer skill for rich HTML output at each phase boundary. You describe what you want to build, and the workflow guides you through understanding the codebase, designing the solution, getting your sign-off, and then executing.

Phase 1

Research

Deep-read the codebase. Produce a markdown research doc and an HTML architecture diagram. You review both before moving on.

Phase 2

Plan + Annotate

Write a detailed implementation plan. You annotate it with corrections. Repeat 1-6 times until you approve. Then a visual plan HTML is generated.

Phase 3

Implement

Execute every task from the plan. Track progress by checking off items. After completion, a fact-checked diff review HTML is generated automatically.

Throughout

Visual Explainer

Generates self-contained HTML pages with Mermaid diagrams, CSS Grid layouts, styled tables. Opens in your browser. No ASCII art.

3 Phase 1 — Research
What happens

The agent reads every file in the target area, traces data flows end-to-end, identifies patterns and edge cases, and writes a comprehensive research.md. Then it generates an architecture.html showing module relationships as a Mermaid diagram or CSS Grid layout, and opens it in your browser.

Your job: Review both artifacts. Does the agent understand the system correctly? If not, say so and the research will be revised. When satisfied, say "move to planning".
research.md includes
  • › System overview
  • › Architecture and component relationships
  • › Key files with roles
  • › Data flow traces
  • › Patterns and conventions
  • › Edge cases and gotchas
architecture.html includes
  • › Module dependency diagram
  • › Data flow visualization
  • › Dark/light theme support
  • › Zoom controls on all diagrams
  • › Staggered animations
  • › Opens automatically in browser
4 Phase 2 — Plan + Annotate
What happens

The agent writes a detailed plan.md with real code snippets, file-by-file changes, testing strategy, and trade-offs. You open it in your editor, add inline annotations, and the agent addresses every note. This cycle repeats until you approve. Then a visual-plan.html is generated with state machines, before/after panels, and edge case tables.

Annotation patterns you can use:
"this should be a PATCH, not a PUT" — correct assumptions
"remove this section entirely" — reject approaches
"the signatures of these functions should not change" — add constraints
"use drizzle:generate for migrations" — provide domain knowledge
No code is written until you approve the plan. This is the most important guard rail. All creative decisions happen here, not during implementation.
5 Phase 3 — Implement
What happens

The agent executes every task from the plan's todo list, checking off items as it goes. It runs type checks and linters continuously. After all tasks are complete, it generates a diff-review.html comparing the working tree against the branch point, then automatically fact-checks the review (correcting any fabricated numbers or function names) before opening it in your browser.

diff-review.html includes
  • › Executive summary with KPI dashboard
  • › Before/after architecture diagrams
  • › Side-by-side code comparisons
  • › Good/Bad/Ugly code review
  • › Decision log with rationale
  • › Verification summary (fact-checked)
After implementation
  • › Terse corrections are fine: "wider", "still cropped"
  • › Agent acts immediately without clarification
  • › If things go wrong, revert and restart fresh
  • › Run /fact-check on any HTML page anytime

8 Quick Start

Invoke the workflow

Type /build-feature add cursor-based pagination to the users API

Review research

Read the research.md and browse the architecture diagram in your browser. Confirm it's correct or point out what's wrong.

Say "move to planning"

The agent writes plan.md. Open it in your editor, add inline notes, and tell the agent to address them. Repeat until satisfied.

Approve the plan

Say "approved" or "looks good". The agent generates a todo list and a visual plan HTML.

Say "implement"

The agent executes every task, checks them off, and produces a fact-checked diff review when done. That's it.

9 Key Principles
Never code before approval

The plan must be explicitly approved. This guard rail prevents wasted work on wrong designs.

Markdown files are deliverables

Not chat summaries. The files in docs/claude/ are the source of truth, readable by anyone.

You inject the judgment

Through annotations: domain knowledge, business constraints, engineering trade-offs. The agent does the work, you steer.

Implementation should be boring

All creative decisions happen during planning. If implementation surprises you, the plan wasn't detailed enough.
