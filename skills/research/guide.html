1 Overview

The research skill performs a deep investigation of a specified codebase area and produces two persistent artifacts: a markdown research document (research.md) and a visual architecture diagram (architecture.html). Together, these form a verified knowledge base for subsequent planning and implementation phases. The user reviews and annotates them before any work begins.

What it does
  • Reads every file in the target area
  • Traces data flows end-to-end
  • Identifies patterns and conventions
  • Maps component relationships
  • Catalogues edge cases and gotchas
  • Documents current state and tech debt
  • Generates a visual architecture diagram
What it does NOT do
  • Change any code
  • Propose solutions or plans
  • Summarize in chat (the file IS the deliverable)
  • Skip files or skim at the signature level
  • Proceed to planning without your review
2 Core Principle
Surface-level reading is NOT acceptable.

You must read deeply, understand intricacies, trace flows end-to-end, and identify edge cases. Do not skim files at the signature level and move on. The difference between a good research document and a useless one is the depth: function signatures vs. actual data flow, config file existence vs. what the config values mean, test file names vs. what behaviors the tests actually verify.

The written artifact is the deliverable — not a verbal summary in chat. The file in docs/claude/ is the source of truth. It should be readable by anyone, standalone, without conversation context.
3 Process — 5 Steps

Identify the scope

Determine what area, module, or topic to research based on $ARGUMENTS. A clear scope prevents unbounded exploration.

Deep-read the code

Thoroughly explore the relevant code. This means:

  • Read every file in the target area, not just entry points
  • Trace data flows from input to output
  • Understand how components interact with each other
  • Identify patterns, conventions, and architectural decisions
  • Note dependencies on other parts of the system
  • Look for edge cases, error handling, and implicit assumptions
  • Examine tests to understand expected behavior
  • Check configuration files for relevant settings

Write the research document

Create research.md at the appropriate path (see File Naming below). The document must include all 8 required sections. Be exhaustive — include specific file paths and line references. Call out anything unclear or buggy explicitly.

Generate visual architecture diagram

After writing the research document, invoke /generate-web-diagram to produce an HTML architecture diagram showing module relationships, data flows, and component boundaries discovered during research.

Output location: Write to architecture.html in the same feature directory as research.md.
Example: docs/claude/20260304-1430-auth-flow/architecture.html
Do NOT write to ~/.agent/diagrams/ or any other location. Open it in the browser.

Stop and wait for review

Tell the user the exact file paths for both research.md and architecture.html, then stop. Invite the user to annotate using the // convention (see below). Do NOT proceed to planning or implementation.

4 Document Sections

Every research.md must include these 8 sections. Completeness matters more than brevity.

Overview

What the system or module does at a high level. 2-3 paragraphs establishing context for everything that follows.

Architecture

How components are organized and relate to each other. Module boundaries, service interfaces, communication patterns.

Key Files

List of important files with their roles. Include file paths, purpose, and notable details about each.

Data Flow

How data moves through the system. Trace request/response cycles, event propagation, state transformations.

Patterns & Conventions

Design patterns used, naming conventions, code style. What a new contributor would need to know to write consistent code.

Dependencies

Internal and external dependencies. What this module relies on, and what relies on it.

Edge Cases & Gotchas

Non-obvious behavior, implicit assumptions, potential pitfalls. The things you only learn by reading the code deeply.

Current State

Known issues, technical debt, areas of concern. Things that should be addressed but haven't been.

6 Annotation Convention

After the research document is delivered, the user reviews it and adds inline corrections using // comments. This is a lightweight way to mark errors, add missing context, or flag things the agent got wrong — without rewriting the document.

Type // followed by your note anywhere in the markdown file. Then tell the agent to address your notes.

## Data Flow

// this actually goes through the message queue first, not directly to the handler Requests are routed directly from the API gateway to the handler...

// wrong — we deprecated this in v2.3, it uses the new adapter now The legacy adapter transforms the response before...
Review prompt delivered to the user: "The research document is ready for your review at <file-path>. I've also generated a visual architecture diagram at <diagram-path> (opened in your browser). To annotate, add // comments anywhere in the file. Just type // followed by your note — corrections, missing context, or things I got wrong. Then tell me to address your notes."
7 Guidelines
Be exhaustive

Better to include too much detail than too little. You can always trim later, but you can't add what you didn't discover.

Use specific references

Include file paths and line numbers. "The auth middleware" is vague; src/middleware/auth.ts:47 is actionable.

Call out unknowns

If something is unclear or seems buggy, say so explicitly. Ambiguity in research becomes wrong assumptions in planning.

No solutions yet

Research documents what IS, not what SHOULD BE. Proposed solutions come during the plan phase, after the user validates your understanding.

No code changes

Do NOT make changes to any code during research. The only files you create are research.md and architecture.html.

The file is the deliverable

The written artifact matters, not a verbal summary in chat. It should be readable standalone, without conversation context.

8 Output

The result is two files in docs/claude/ that serve as a verified knowledge base for subsequent planning and implementation:

research.md

Comprehensive written research with all 8 sections. The primary deliverable. Reviewed and annotated by the user using // comments.

architecture.html

Visual architecture diagram showing module relationships, data flows, and component boundaries. Opens in the browser for quick spatial understanding.

What happens next: The user reviews both artifacts, adds // annotations to correct any misunderstandings, then confirms the research is accurate. Only then does the workflow proceed to planning with /plan.
